## SECTION 1 — Scope + Definitions

### 1.1 Scope

This contract governs the **user experience layer** of the TODAY screen. It defines:
- What UI surfaces exist and their exact purpose
- What content appears in each surface
- How resolved state maps to visual output
- What user actions are permitted or forbidden
- How truthfulness is maintained in display

This contract does NOT govern:
- State resolution logic (see CONTRACT_2.2)
- Merlin dialogue content (see CONTRACT_2.3)
- Coach action mechanics (see CONTRACT_2.4)
- Backend authorization (see Authorization & Guardrails Contract v1)

### 1.2 Definitions

#### TODAY Screen
The primary execution interface displayed when an athlete opens the application. TODAY shows training content for the **current calendar day only**. TODAY is the execution surface — not strategy, not planning, not history.

#### Plan
The resolved training session for today. A plan consists of ordered blocks that the athlete executes sequentially. A plan is authored by coach/program and resolved by the backend. A plan is **never** generated by the UI or AI at runtime.

#### Block
A discrete unit of training activity within a plan. Blocks have a defined type, content, and sequence position. Block types include: morning_mobility, foam_roll, warm_up, main_session, cool_down, recovery, pre_practice_activation, rehab_exercises, film_room, manual_session_log.

#### Banner
A notification surface displayed at the top of the TODAY screen. Banners communicate state context, warnings, alerts, or errors. Banners have defined types with strict visual hierarchy.

#### Override
A condition that replaces or modifies the default training plan. Overrides are set by coach action, system safety triggers, or external scheduling (practice, weather, taper, rehab). Overrides MUST be visible before training blocks.

#### coach_locked
A session flag indicating that the coach has modified the session. When `coach_locked = true`:
- AI personalization MUST NOT modify the session
- Athlete requests to modify MUST be refused
- Coach attribution MUST be displayed
- Session executes exactly as coach specified

#### Acknowledgment Gate
A blocking UI overlay that requires explicit athlete action before proceeding. Acknowledgment gates are used for coach alerts, safety warnings, and critical notifications. Gates MUST NOT be dismissible via swipe, back navigation, or timeout.

## SECTION 2 — Single-Plan Rule

### 2.1 Core Principle

The TODAY screen MUST display **exactly one execution plan** for the current day.

### 2.2 Binding Rules

1. **ONE Plan Only:** TODAY MUST NOT display multiple plan options, alternatives, or "choose your workout" interfaces.

2. **No Plan Generation:** If the backend cannot resolve a valid plan, TODAY MUST display an error state. TODAY MUST NOT generate, suggest, or improvise training content.

3. **No Plan Switching:** TODAY MUST NOT provide toggles, tabs, or controls to switch between plan variants (e.g., "Easy Day" vs "Full Session").

4. **No AI Alternatives:** Merlin MUST NOT suggest alternative sessions. If athlete asks for alternatives, Merlin MUST refuse and direct to coach.

5. **Decision Finality:** The plan displayed is the plan to execute. The decision has been made by coach/program/system. TODAY is execution, not strategy.

### 2.3 Forbidden Patterns

- "Option A: Program workout" / "Option B: Recovery day"
- "Choose your session" picker
- "Switch to taper plan" toggle (when taper not active)
- "Try this instead" suggestions
- "Easier option" / "Harder option" buttons
- AI-generated session alternatives

## SECTION 3 — Timeframe Rule

### 3.1 Core Principle

The TODAY screen MUST display content for **today only** — the current calendar day in the athlete's local timezone.

### 3.2 Binding Rules

1. **Today Only:** TODAY MUST NOT display tomorrow's session, next week's plan, or any future content.

2. **No Date Picker:** TODAY MUST NOT include a date picker, calendar selector, or any control that allows viewing other days.

3. **No Future Preview:** TODAY MUST NOT include "Tomorrow: [Session Type]", "Next 7 days", "Upcoming this week", or similar preview sections.

4. **No Past Sessions:** TODAY MUST NOT display yesterday's session or historical training data. History belongs in the History/Calendar view.

5. **Timezone Authority:** "Today" is determined by the athlete's device timezone. If timezone changes mid-session, TODAY MUST NOT retroactively alter displayed content.

### 3.3 Rationale

Mixing timeframes creates confusion about what to execute now. The Calendar/Week view exists for planning context. TODAY exists for immediate execution clarity.

## SECTION 4 — Authority Surfaces

### 4.1 Surface Hierarchy (Top to Bottom)

The TODAY screen MUST display content in this exact visual hierarchy:

1. **Header Area** — Date, athlete name, readiness indicator (if available)
2. **Banner Area** — Alerts, warnings, info banners (stacked by priority)
3. **Coach Attribution Area** — Coach name, modification timestamp, override type
4. **Coach Notes Area** — Verbatim coach message (if present)
5. **Blocks Area** — Ordered training blocks
6. **Action Area** — Primary and secondary CTAs
7. **Merlin Access** — Entry point to AI assistant (non-intrusive)

### 4.2 Coach Attribution Display Rules

When a session has been modified by a coach (`coach_locked = true` or coach action present):

**Required Display:**
```
Updated by Coach [Full Name] at [Time] on [Date]
```

**Placement:** Above all training blocks, below banners

**Persistence:**
- MUST remain visible until athlete acknowledges (if acknowledgment required)
- MUST remain visible for 7 days after session completion
- MUST NOT be dismissible by athlete

**Multiple Modifications:**
- Show only most recent change on TODAY
- Display count: "Updated 3 times by Coach [Name]"
- Provide "View Change Log" link for full history

### 4.3 Coach Notes Display Rules

When coach has attached a note to the session:

**Required Display:**
```
📝 Coach Note (Priority: [info|attention|urgent])
From Coach [Name] • [Date] at [Time]
[Note content verbatim — no paraphrasing, no truncation]
```

**Placement:** Below coach attribution, above training blocks

**Visual Treatment:**
- Distinct container (border, background color)
- Priority indicator visible
- Timestamp visible

**Verbatim Requirement:** Coach notes MUST be displayed exactly as written. The UI MUST NOT summarize, shorten, or reinterpret coach notes.

### 4.4 Override Display Rules

When an override is active (weather, practice, taper, rehab, film_room):

**Required Display:**
- Override type icon
- Override reason
- Original plan indicator (dimmed): "Original: [Session Type] • [Duration]"
- Modified plan indicator (primary): "Coach Update: [New Session Type] • [New Duration]"

**Placement:** In banner area AND in coach attribution area

## SECTION 5 — Resolution Binding

### 5.1 Core Principle

The UI MUST faithfully render the `TodayViewModel` output from the backend. The UI MUST NOT re-decide, reinterpret, or modify resolved state.

### 5.2 Binding Rules

1. **UI Reflects Resolution:** Every field in `TodayViewModel` MUST map to exactly one UI element. No field is optional for display.

2. **No UI-Side Logic:** The UI MUST NOT contain conditional logic that alters what the backend resolved. If backend says "show warning banner", UI shows warning banner.

3. **No Fallbacks:** If `TodayViewModel` contains unexpected/invalid data, UI MUST display error state. UI MUST NOT substitute placeholder content.

4. **Deterministic Rendering:** Given identical `TodayViewModel`, the UI MUST produce identical visual output across all platforms, all users, all runs.

5. **No Caching Staleness:** UI MUST request fresh `TodayViewModel` on every TODAY screen open. Cached data MUST NOT be displayed if >5 minutes old.

### 5.3 TodayViewModel → UI Mapping (Required)

| ViewModel Field | UI Element | Display Rule |
|-----------------|------------|--------------|
| `training_allowed` | Blocks visibility | If false, show error state only |
| `main_session_source` | Session title | Display source name |
| `blocks_shown` | Block list | Render in exact order |
| `banner_type` | Banner component | Use correct type styling |
| `banner_text` | Banner content | Display verbatim |
| `primary_cta` | Primary button | Display with correct label |
| `secondary_cta` | Secondary button | Display if not null |
| `merlin_posture` | Merlin availability | Enable/disable based on posture |
| `coach_attribution` | Attribution area | Display if present |
| `coach_notes` | Notes area | Display verbatim if present |
| `readiness_score` | Header indicator | Display if not null, else "—" |
| `acwr_baseline_progress` | Baseline widget | Display X/21 if building |

## SECTION 6 — Banner System

### 6.1 Banner Types (Canonical)

| Type | Color | Icon | Use Case |
|------|-------|------|----------|
| Error | Red | ⛔ | Session resolution failure, no program, system error |
| Alert | Amber | ⚠️ | Rehab protocol, coach alert, weather override, ACWR danger |
| Warning | Yellow | ⚡ | Stale readiness, approaching ACWR threshold |
| Info | Blue | ℹ️ | Practice day, film room, taper, missing check-in, baseline building |

### 6.2 Stacking Order (Hard Priority)

When multiple banners apply, display in this order (top to bottom):

1. **Error** (highest priority)
2. **Alert**
3. **Warning**
4. **Info** (lowest priority)

Within the same type, stack by timestamp (most recent first).

### 6.3 Maximum Banners

TODAY MUST NOT display more than **3 banners** simultaneously. If more than 3 conditions exist:
- Display top 3 by priority
- Provide "View all alerts" link to expand

### 6.4 Dismissal Rules

| Type | Dismissible | Persistence |
|------|-------------|-------------|
| Error | NO | Until condition resolved |
| Alert | Only after acknowledgment | Until acknowledged |
| Warning | YES (swipe) | Returns on next TODAY open if condition persists |
| Info | YES (swipe) | Returns on next TODAY open if condition persists |

### 6.5 Banner Text Requirements

- Banner text MUST be factual and specific
- Banner text MUST state consequence or required action
- Banner text MUST NOT use guilt language ("You forgot to...")
- Banner text MUST NOT use pressure language ("You must...")
- Banner text MUST NOT use vague language ("Something may be wrong")

## SECTION 7 — Blocks System

### 7.1 Block Ordering Rules

Blocks MUST appear in the following canonical order:

1. Morning mobility
2. Foam roll
3. Warm-up / Pre-practice activation
4. Main session / Rehab exercises / Film room / Manual session log
5. Cool-down
6. Recovery

This order MUST NOT be changed by user preference, AI suggestion, or frontend logic.

### 7.2 Mandatory Blocks

The following blocks MUST appear when their conditions are met:

| Condition | Mandatory Blocks |
|-----------|------------------|
| Normal training day | Morning mobility, foam roll, warm-up, main session, cool-down, recovery |
| Practice day | Morning mobility, foam roll, pre-practice activation, [practice placeholder], post-practice recovery |
| Rehab protocol active | Morning mobility (injury-safe), rehab exercises, recovery |
| Film room day | Morning mobility, foam roll, film room, optional skill drills, recovery |
| External program | Morning mobility, foam roll, manual session log area, recovery |

### 7.3 Block Removal Rules

A block MUST NOT be removed from display without explanation.

**When block is removed:**
- Display placeholder: "[Block Name] not included today"
- Display reason: "Reason: [specific explanation]"
- Log removal in session view log

**Examples:**
- "Warm-up not included today. Reason: Practice prep replaces standard warm-up."
- "Main session not included today. Reason: Film room day — no field training scheduled."

### 7.4 Block Content Requirements

Each block MUST display:
- Block title
- Estimated duration
- Exercise count (if applicable)
- Completion status (not started / in progress / complete)
- Expand/collapse control (for exercise detail)

## SECTION 8 — Coach Alert Gate

### 8.1 Triggering Conditions

A Coach Alert Gate MUST be displayed when:
- `coachAlertActive === true`
- `coachAlertPriority === "high"` or `coachAlertType === "session_modified"` with acknowledgment required
- Coach override exceeds ACWR safety boundary
- Coach proceeds against weather cancellation recommendation

### 8.2 Gate Display Rules

**Visual Treatment:**
- Full-screen overlay (semi-transparent background)
- Modal container centered
- Cannot be dismissed via background tap
- Cannot be dismissed via back navigation
- Cannot be dismissed via swipe

**Required Content:**
```
🔔 Coach Alert

[Coach Message — verbatim, no truncation]

From Coach [Name] • [Date] at [Time]

[Read Coach Message] [Acknowledge]
```

### 8.3 Acknowledgment Requirements

1. **Read First:** Athlete MUST scroll through full message before "Acknowledge" button becomes active (if message >200 characters)

2. **Explicit Action:** Acknowledgment requires button press labeled "Acknowledge" or "I Understand"

3. **No Auto-Acknowledge:** System MUST NOT auto-acknowledge based on time viewed or screen open duration

4. **No Bypass:** Training blocks MUST NOT be accessible until acknowledgment complete

### 8.4 Acknowledgment Persistence

Once acknowledged:
- Gate closes
- Training blocks become accessible
- Acknowledgment timestamp logged
- Acknowledgment CANNOT be undone
- Coach notified of acknowledgment

## SECTION 9 — Truthfulness Requirements

### 9.1 Null Readiness Display

When `readinessScore === null`:

**MUST Display:**
- Empty indicator: "—" or "Not logged"
- Neutral language: "Check-in not logged yet"

**MUST NOT Display:**
- Any numeric value (0, 50, 75, "average")
- Placeholder: "Readiness: —/100"
- Imputed value from other metrics
- "Low readiness" or any interpretation

### 9.2 ACWR Baseline Display

When `confidenceMetadata.acwr.confidence === "building_baseline"`:

**MUST Display:**
- Progress format: "X/21 training days logged"
- Progress language: "Building your load baseline"
- Milestone markers at 7, 14, 21 days

**MUST NOT Display:**
- "Low confidence"
- "Insufficient data"
- "Not enough history"
- "Unreliable"
- ACWR ratio value (until baseline complete)
- Injury risk indicators (until baseline complete)

### 9.3 Stale Data Display

When data is stale (>24 hours old):

**MUST Display:**
- Age indicator: "Last updated X days ago"
- Source timestamp: "[Value] from [Date]"
- Consequence: "Plan uses program defaults"

**MUST NOT Display:**
- Stale data without age indicator
- Stale data styled as current
- "Outdated" or "old" (use "last updated")

### 9.4 Missing Data Display

When any required data is missing:

**MUST Display:**
- Explicit absence: "Not available"
- Reason if known: "Requires check-in"
- Action if applicable: "Log check-in to see readiness"

**MUST NOT Display:**
- Empty space with no explanation
- Placeholder that could be mistaken for real data
- "Error" without context

## SECTION 11 — Forbidden UX Patterns

### 11.1 Patterns Inherited from Resolution Contract (12 Bans)

The following patterns are FORBIDDEN per TODAY State → Behavior Resolution Contract v1:

1. **Showing Multiple Plans Simultaneously** — No "Option A / Option B" interfaces
2. **Suggesting Alternatives on TODAY** — No "Can't do squats? Try goblet squats instead"
3. **Optimistic Language During Rehab** — No "You'll be back soon!"
4. **"Low Confidence" Wording for Baseline ACWR** — No "Low confidence ACWR"
5. **Hiding Coach Overrides** — No silent modifications
6. **Generating Replacement Workouts** — No AI-generated fallbacks
7. **Silent Intensity Changes** — No invisible volume adjustments
8. **Fake Readiness Values** — No showing "75" when null
9. **Skipping Blocks Without Explanation** — No silent removal
10. **Letting Merlin Negotiate Safety** — No "Let me check if we can add this"
11. **Showing Future Sessions on TODAY** — No "Tomorrow: Lower Body"
12. **Blocking Training for Non-Safety Reasons** — No "Complete check-in to unlock training"

### 11.2 Additional UX Bans (Extended)

13. **Gamification of Compliance** — No streaks, badges, or rewards displayed on TODAY that pressure behavior

14. **Social Comparison** — No "Your teammates have completed check-ins" or leaderboards

15. **Artificial Urgency** — No countdown timers, "Last chance!", or flashing elements

16. **Guilt-Based Prompts** — No "You forgot to...", "You should have...", "Don't you want to..."

17. **Nagging Repetition** — No repeated prompts for same action within single session

18. **Deceptive CTAs** — No buttons that do something other than their label suggests

19. **Hidden Costs** — No actions that have undisclosed consequences

20. **Manipulative Defaults** — No pre-selected options that benefit system over athlete

21. **Confusing Hierarchy** — No visual design that obscures what to do next

22. **Information Overload** — No more than 3 banners, no more than 6 visible blocks without scrolling

23. **Inconsistent Terminology** — Same concept MUST use same term everywhere (e.g., "readiness" not "wellness score" interchangeably)

24. **Ambiguous States** — Every UI state MUST have clear visual treatment; no "is this loading or empty?"

## SECTION 12 — Acceptance Criteria + QA Checklist

### 12.1 Deterministic Output Criteria

Given identical `TodayViewModel` input, the TODAY screen MUST produce:

- [ ] Identical banner display (type, text, order)
- [ ] Identical block display (type, order, content)
- [ ] Identical CTA display (label, enabled state)
- [ ] Identical coach attribution (if present)
- [ ] Identical readiness indicator (value or "—")
- [ ] Identical ACWR baseline indicator (X/21 or hidden)

### 12.2 Cross-Platform Parity Criteria

The following MUST be identical across web, iOS, and Android:

- [ ] Banner stacking order
- [ ] Block ordering
- [ ] Coach attribution format
- [ ] Acknowledgment gate behavior
- [ ] Truthfulness display (null handling)
- [ ] Forbidden pattern prevention

| Test Case | Expected Result | Pass/Fail |
