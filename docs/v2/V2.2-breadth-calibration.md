# V2.2 — Breadth & Calibration

**Status: PARTIAL.** One real slice landed (QB throw-count logger). Everything else in this phase — coach tournament console, session library breadth, monotony/strain, sweat-rate calibration, coach-tunable engine constants — is **specced below, not built**. This doc exists so the spec survives to whoever picks the phase up next, without pretending it's done.

## What's built

### QB arm-care / throw-count logger
The V1 audit flagged `netlify/functions/qb-throwing.js` as ORPHANED: a complete, tested backend (`qb_throwing_sessions` table, `GET`/`POST /api/qb-throwing`, a progression RPC, weekly stats aggregation, 424 lines) with **zero frontend callers**. The engine's `QB_THROW_ADAPTATION` dosing policy (`position-volume.config.ts`) has always assumed throw-count data exists to gate on — it never did.

V2.2 wires a minimal logger:
- `angular/src/app/core/services/qb-throwing.service.ts` — thin client for the existing endpoint (load progression, log a session).
- `angular/src/app/shared/qb-arm-care-card.component.ts` — a self-gating card (renders nothing unless `PeriodizationService.position() === 'qb'`) showing this week's throw progression + a quick log form (session type, total throws, arm feeling after).
- Dropped into the Training screen (`<app-qb-arm-care-card />`) — any other screen can include it without its own position check.

No new migration — this phase only adds a frontend caller for infrastructure that already existed. Verified: `tsc --noEmit` clean, `ng build` (AOT) clean, ESLint clean, full Angular suite (374 tests) passes with zero regressions.

**Still open:** the short/medium/long throw-type split, warmup/arm-care/ice checkboxes, and session history view exist on the backend and in the data model but aren't in the quick-log form yet (kept minimal for a first wire-up). The `QB_THROW_ADAPTATION` dosing policy itself doesn't yet read this data — it's specced to reduce throw volume on a high interception rate, which isn't logged anywhere; the throw-count data this phase adds is a prerequisite, not the full loop.

## Specced, not built

### Coach tournament console
Day board (team's game timeline × roster readiness heat-strip), rotation/minutes-exposure hints, one-tap re-time propagating to every athlete's timeline. Depends on V2.0's `event_games` (built) + the per-game participation logging that's still open from V2.0 (§ "Explicitly not built" in `V2.0-tournament-mode.md`). Natural next step: a `staff/gameday-console/` screen reading `event_games` + `event_participation` team-scoped, reusing `coach-core.js`'s existing consent-gated readiness read.

### Session library breadth
Flag-specific speed/agility day, tempo/conditioning day, plyometric progression, hotel/no-equipment variants, micro-dosing days. All of these are content additions to the existing COMPOSE path (`daily-protocol.js` intent → realized exercises) — the architecture doesn't need to change, only the exercise library and intent catalogue. Not started; would need exercise-library content work more than engineering.

### Monotony & strain (Foster)
Weekly load/SD and load×monotony alongside ACWR, computed from the same `training_sessions` data `utils/acwr.js` already reads. A pure-function addition parallel to `computeAcwr` — not started.

### Sweat-rate calibration
Pre/post weigh-in option on tournament days → per-athlete sweat rate stored and reused for future heat plans, replacing the fixed ml/kg heuristic. Would need a new small table + a weigh-in prompt on the gameday screen. Not started.

### Coach-tunable engine constants
Surfacing the documented calibration constants (350 AU/game estimate, `CARB_PER_KG` table, congestion thresholds, travel penalty, the new V2.1 caffeine constants) in a staff settings screen, writing to the already-live `prescription_templates`/`taper_rules` tables. Not started — these tables exist live (per `SOURCE_OF_TRUTH.md` §4) with no UI.
