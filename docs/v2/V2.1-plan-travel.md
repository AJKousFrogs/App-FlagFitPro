# V2.1 — Travel, Supplement Timing, Season Plan

**Status: PARTIAL.** Two real, tested slices landed (travel legs + wellness suggestion; caffeine timing). The season plan builder and knowledge-base contextual cards from the V2 proposal (§7, §8) are **not built** — this doc says so explicitly rather than leaving the gap implied. Same live-DB caveat as V2.0: `event_travel`'s migration is committed, not applied.

## What's built

### `event_travel` (`supabase/migrations/20260703030000_create_event_travel.sql`)
Athlete-declared travel legs (`mode`, `depart_at`, `arrive_at`, `timezone_delta_hours`, `overnight_stay`), optionally linked to a `competition_event` so a coach-arranged team trip is visible to team staff too. Owner-scoped RLS (mirrors `athlete_events`), `team_id` trigger-synced when linked (mirrors `event_games`'s pattern from V2.0).

### Backend (`netlify/functions/event-travel.js`)
Full CRUD (`GET`/`POST`/`PATCH`/`DELETE /api/event-travel`), owner-scoped. 10 integration tests (`tests/integration/event-travel-validation.test.js`), all passing.

### Frontend: travel-aware wellness suggestion
`EventTravelService` (`angular/src/app/core/services/event-travel.service.ts`) loads legs and exposes `legToday`/`todayTravelHours`. The wellness check-in (`wellness.component.ts`) shows a suggestion banner — **"You declared a trip today (~Xh) — apply it?"** — when a declared leg covers today AND the travel-hours chip is still at its untouched default. The athlete taps Apply; nothing is silently set. This deliberately stays a *suggestion*, not a silent prefill, to respect the daily-input-forms law in `SOURCE_OF_TRUTH.md` §5b (never let a default look like a real entry) — a same-day leg existing doesn't by itself prove the athlete actually traveled that many hours today (a leg can span midnight, get shortened, etc.), so it's offered, not asserted.

The reactive backstop — `daily_wellness_checkin.travel_hours` → the bounded readiness penalty in `calc-readiness.js` — is **unchanged**. This is additive: a proactive nudge in front of the existing reactive path, not a replacement.

### Frontend: game-day caffeine timing (`supplements.component.ts`)
On a competition-phase day, a new card computes:
- Suggested dose: 3 mg/kg (bodyweight from `user_metadata`, same mirrored lookup pattern as `tournament-plan.service.ts`), taken ~50 min before kickoff. Kickoff time comes from V2.0's `event_games` when the coach entered per-game times, else the day-level event.
- **Sleep-cutoff guard**: if the last game of the day ends within 8 hours of an assumed 23:00 bedtime, a warning tells the athlete to skip a late top-up — sleep matters more than one more caffeine edge before a knockout game. The 8h/23:00 constants are documented in-code as starting points (no per-athlete bedtime setting exists yet — flagged as a real simplifying assumption, not fabricated personalization).

Verified: `tsc --noEmit` clean, `ng build` (AOT) clean, ESLint clean, full Angular suite (374 tests, incl. new) + full backend suite (595 tests) all pass with zero regressions.

## Explicitly not built (do not assume these exist)

- **Season plan builder** (V2 proposal §8) — `team_season_phases` table exists live (per `SOURCE_OF_TRUTH.md` §6, flagged DRIFT) but has no reverse-periodization UI, no event A/B/C priority column, no mesocycle skeleton generation. Not started.
- **Knowledge-base contextual cards / tournament content pack / sl-SI localization** (§7) — no `knowledge_context_tags`, no `language` column, no new KB entries added this phase.
- **In-transit micro-mobility content, jet-lag protocol card, arrival-day load capping** — `event_travel.timezone_delta_hours` is captured in the schema for this but no jet-lag threshold logic reads it yet (documented on the column comment as a V2.2+ hook).
- **Travel-leg entry UI** — legs can only be created via the API right now; no in-app form to declare one (the wellness suggestion only *reads* legs). A coach- or athlete-facing "add a trip" screen is the natural next slice.
- **Live-DB application** of `20260703030000_create_event_travel.sql` — same caveat as V2.0.
