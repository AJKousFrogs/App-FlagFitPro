## The brief

> Players from American Samoa to Mongolia ‚Äî December in Portugal is not December in Slovenia. Some players have continentals every year, some Worlds every 2 years, some the Olympics, some travel internationally for tournaments. Everything has to be considered.

Two real gaps, both now closed:

1. **Weather/climate handling was already global** (Open-Meteo, real geocoded lat/lon, `timezone=auto`) ‚Äî but treated every athlete identically regardless of whether they'd just landed somewhere. A Ljubljana athlete arriving in American Samoa yesterday faces materially higher heat-illness risk at 32¬∞C than a Samoan native; a Samoan arriving in Mongolia in January faces the same asymmetry in reverse. **Nothing modeled acclimatization.**
2. **The competition calendar had no tier.** A Continental Championship (most players have one every year), a World Championship (every ~2 years), and the Olympics (every 4) all collapsed onto the same generic "national team event" ‚Äî the taper/recovery engine only read a coach-set `importance` flag that could be forgotten, with no guaranteed floor for how big the occasion actually is.

## 1. Competition-tier-aware taper/recovery

`competitions.level` (`club`/`regional`/`national`/`international`/`continental`/`world`/`olympic`) has existed in the schema since the original schedule-spine migration ‚Äî **nothing read it**. The periodization phase resolver (`resolvePhase`, mirrored in `netlify/functions/schedule.js` and `angular/.../schedule.service.ts`) only branched on the coach/athlete-set `importance` flag (regular/high/peak).

**Fix:** `effectiveImportance(importance, competitionLevel)` ‚Äî the higher of the declared importance and a tier floor (`international`/`continental` ‚Üí floor `high`; `world`/`olympic` ‚Üí floor `peak`; never lowers a declared importance). `world` and `olympic` additionally extend the taper/recovery window itself (+3 days taper / +1 day recovery for `world`; +7 days taper / +3 days recovery for `olympic`) on top of the floor ‚Äî a Continental gets standard "peak" treatment, but a World Championship or the Olympics taper and recover measurably deeper, regardless of whether anyone remembered to flag it.

**New: `athlete_events.tier`.** Athlete-entered national-team commitments (`category: "national"`) previously all mapped to a flat `"international"` level ‚Äî a routine camp, a Continental Championship, a World Championship, and the Olympics were indistinguishable to the engine. `tier` (`continental`/`world`/`olympic`, nullable) overrides the category-based default when set. Wired end-to-end: migration ‚Üí `athlete-events.js` validation ‚Üí `athlete-event.models.ts` ‚Üí the Schedule screen's add/edit form (a tier chip row appears when "National team" is selected) ‚Üí the event list badge shows "World Championship" instead of the generic "National team" label.

**Team-scheduled events (`competition_events`) can already carry continental/world/olympic** via `competitions.level` ‚Äî the engine reads it correctly ‚Äî **but no UI creates a `competitions`/`competition_events` row at all** (confirmed: zero endpoint references either table for writes). Setting a team event's tier today requires direct DB access. A coach-facing competition/event creation screen is the natural next step and is explicitly **not built**.

Tests: 12 new cases in `tests/unit/schedule-resolver.test.js` (server, 38 total) + a new `schedule.service.spec.ts` (client, 10 cases) ‚Äî both suites cover the same scenarios so the client/server mirror-contract this repo insists on is actually verified on both sides, not just documented as a requirement.

## 2. Heat/cold acclimatization guard

`applyWeatherGuard` (the existing heat/cold/storm/wind guard) gained a 4th parameter, `acclimatizationDay` (days since the athlete's most recent arrival at their current location). Rather than requiring a computed "home climate baseline" the app doesn't have, **every threshold shifts symmetrically** ‚Äî heat thresholds down, cold thresholds up ‚Äî while the athlete is within a 14-day acclimatization window, decaying linearly to zero (byte-identical to pre-V2.4 behavior) by day 14. One mechanism correctly protects both directions: a hot-destination arrival (American Samoa) and a cold-destination arrival (Mongolia in January) without needing to know which direction the trip went.

`EventTravelService.daysSinceArrival` computes this from the real `arriveAt` timestamp of the most recent past-arrival leg ‚Äî **not** the DB's static `adaptationDay` field, which would silently go stale if an athlete didn't update it daily. `weekAhead()` extrapolates it forward (`day + i`) since, unlike weather/ACWR, it's simple arithmetic rather than an unforecastable current-moment signal.

Tests: 5 new cases in `periodization.service.spec.ts` (99 ‚Üí the full suite grew to 106 across this phase), covering both the heat and cold direction, the day-14 decay boundary, and that `null`/omitted is a no-op.

## 3. Arrival-day load cap

Closes the gap `docs/v2/V2.1-plan-travel.md` explicitly flagged as not built: a same-day arrival ‚â•3h caps the session to activation-only (`mobility` intent, RPE ‚â§4, ‚â§30min, no sprints/sets) ‚Äî the travel itself is a fatigue cost the base plan doesn't otherwise know about. Exempt: `rest`/`recovery`/`mobility` (already at/below the cap), `taper-prime` (already minimal by design), `competition` (game day is the organiser's call, not something to reschedule around). Runs after the weather guard (a thunderstorm's "stop" is more restrictive and wins outright) and before the injury/physio guard, which stays the highest-precedence override per spec law.

`EventTravelService.arrivalDayTravelHours` ‚Äî duration of the leg that arrived specifically _today_ (distinct from `todayTravelHours`, which is "currently mid-transit", and `daysSinceArrival`, which is "how long since any past arrival").

Tests: 7 new cases in `periodization.service.spec.ts`, including the exemption boundaries and the weather-guard precedence order.

## Explicitly not built (do not assume these exist)

- **Coach-facing competition/event creation UI.** `competitions`/`competition_events` have zero write endpoints ‚Äî a team event's tier can only be set via direct DB access today. The tier-aware engine is ready; the entry point isn't.
- **Jet-lag protocol content** (light exposure timing, meal anchoring). `athlete_travel_log.timezone_difference` is captured and documented as a hook for this; no logic reads it yet.
- **Sweat-rate / true heat-acclimatization physiological tracking** ‚Äî the guard is a documented, evidence-based _heuristic_ (symmetric threshold shift decaying over 14 days), not a personalized model. Flagged as a calibration constant, same convention as the rest of the engine's tunables.
- **`tournament_day_plans` / `warmup_protocols` cleanup** ‚Äî identified as dead scaffolding, not removed. A future session with fuller context on their history should decide whether to drop them.
- **`team_events` documentation** ‚Äî found to be a live, wired, but undocumented feature (referenced by `team-calendar.js`, `attendance.js`, `coach-core.js`). Out of scope for V2.4; noted for whoever next touches the Feature Status Ledger.
