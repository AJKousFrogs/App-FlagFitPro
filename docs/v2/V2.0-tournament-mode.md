## What this solves

V1 knew a tournament day only as a count (`competition_events.expected_game_count`). It could not tell the ~50-minute turnaround after a Capital Bowl game 1 (11:00) apart from the ~2h20 gap after game 2 (12:30 → 15:30) — the gameday screen gave the same static "12% of daily carbs between games" note for every gap. V2.0 replaces that with real per-game kickoff times and a generated, gap-classified timeline.

## What's built

### Database (`supabase/migrations/20260702120000_create_event_games.sql`)
- `event_games` — one row per game, child of `competition_events` (`competition_event_id` FK). Columns: `game_number`, `game_date`, `kickoff_time`, `expected_duration_minutes` (default 40), `opponent`, `field`, `bracket_stage`, `is_provisional`, `status`, `result`.
- `team_id` is trigger-synced from the parent `competition_events` row (`event_games_sync_team_id`) — never trusted from the client, can't drift or be spoofed to a different team.
- RLS: SELECT = `ff_is_active_team_member(team_id)`; INSERT/UPDATE/DELETE = `ff_is_team_staff(team_id)` — same helper functions the rest of the schedule spine uses.
- `event_participation.game_id` (nullable FK → `event_games`) — added so a future per-game quick-log can attribute minutes/RPE to a specific game instead of only the day total. **Column exists; no writer uses it yet** (see "Explicitly not built" below).

### Backend (`netlify/functions/event-games.js`, redirects in `netlify.toml`)
- `GET /api/event-games?competitionEventId=` — list games for an event (any active team member).
- `POST /api/event-games` — create one game (staff only).
- `POST /api/event-games/bulk` — the real coach workflow: replace an event's whole game list from an ordered array in one call ("paste 11:00, 12:30, 15:30, 17:00").
- `PATCH /api/event-games/:id`, `DELETE /api/event-games/:id` — re-time or remove a game (staff only, checked in JS via `isStaffOfTeam` in addition to RLS — service-role client bypasses RLS, so the JS check is the real gate).
- Tests: `tests/integration/event-games-validation.test.js`, 14 cases (validation, auth, bulk replace, CRUD) — all passing.

### Frontend gap-classification engine (`angular/src/app/core/services/tournament-plan.service.ts`)
Pure functions, spec-tested the same way `periodization.service.ts` is (`tournament-plan.service.spec.ts`, 23 cases anchored on the actual Capital Bowl day-1 schedule: 11:00/12:30/15:30/17:00, 40-min games):

| Effective gap | Class | Fueling | Warm-up |
|---|---|---|---|
| < 30 min | `turnaround` | fluids + optional half gel | stay warm, no re-warm-up |
| 30–75 min | `short` | ~0.4 g/kg fast carbs + fluid | 8–10 min re-prime |
| 75–150 min | `medium` | ~1 g/kg light solid carbs, ≥60min pre-kickoff | 10–12 min re-warm-up |
| > 150 min | `long` | ~1.25 g/kg + protein meal, ≥75min pre-kickoff | 12–15 min near-full warm-up |

`buildTournamentDayPlan` sequences a full day: wake → breakfast → arrival → full pre-game warm-up → (game → fuel → warm-up)* → final game → recovery block. It applies a heat modifier (mirrors `periodization.service.ts`'s 28°C apparent-temp threshold — documented as a mirror, not re-exported, to avoid touching the 94-test periodization spec contract) and surfaces the existing `TOURNAMENT_DAY.lateGameWarning` hamstring cue on the last game and its lead-in warm-up.

This is presentation/sequencing logic layered on top of the server-canonical prescription (nutrition macros, readiness, ACWR) — it never recomputes those, only sequences a multi-game day around the numbers the engine already produces.

### UI (`angular/src/app/gameday/`)
- When the coach has entered 2+ games for the next event, the gameday screen's fueling section is replaced by the generated timeline (ordered blocks, current/next block highlighted, updates every 60s).
- Falls back to the V1 static day-level fueling card when no per-game kickoff times exist yet — **nothing regresses for events without Tournament Mode data**.
- Coach-only (`TeamMembershipService.isCoach()`) inline editor: a comma-separated kickoff-time input that calls the bulk-set endpoint. Shows "No per-game kickoff times yet" / "N games entered" so the coach knows the state.
- Verified: `ng build` (AOT, catches template binding errors) succeeds; `tsc --noEmit` clean; ESLint clean on all new/changed files; all Angular spec tests pass (312 total, including the 23 new).

## Explicitly not built (do not assume these exist)

- **Per-game actuals / quick-log.** `event_participation.game_id` column exists; no endpoint or UI writes it yet. ACWR still uses the flat ~350 AU/game estimate for tournament days, not real per-game logged load. This is the highest-value V2.0 follow-up — it closes the calibration loop described in the V2 proposal §3.4.
- **Kickoff-timed push notifications.** The proposal's "T−40 start warm-up" notification is not wired — the timeline is pull (open the app), not push, in this build.
- **Offline queue.** No offline-first outbound queue for venue connectivity — flagged as a P0 UX risk in the proposal (§11.1) and still open.
- **Athlete-entered events (`athlete_events`) per-game timing.** Scoped to `competition_events` (team-scheduled) only — see the migration's header comment for the reasoning.
- **Live-DB application** — see the caveat above.
